## 一、总体架构

### 1.1 核心玩法流程

```
┌─────────────────────────────────────────────────┐
│  关卡设计（创造模式）                              │
│                                                 │
│  LevelEditorBlock (256³ 可建造区域)              │
│  ├── 地形搭建（方块 + 红门 + 蓝门 + 控制器）      │
│  ├── 门 ID 设置（手持工具右键门）                 │
│  ├── 节点图编辑（潜行右键控制器）                  │
│  └── 保存为 StructureTemplate NBT                │
│       ├── ModularDefendMeta (关卡名/作者/尺寸)    │
│       │   └── 门 ID → 坐标映射                   │
│       └── NodeGraphData (控制器图数据)            │
└─────────────────────────────────────────────────┘
                       ↓
            config/modularization_defend/levels/
                       ↓  启动时扫描
┌─────────────────────────────────────────────────┐
│  战前大厅（生存/冒险模式）                        │
│                                                 │
│  CombatConsoleBlock (2格高多方块)               │
│  ├── 关卡列表 + 模式选择                          │
│  │   (危机合约 / 探索模式 / 联合作战)             │
│  ├── 词条配置（全局池 + 关卡覆盖）               │
│  ├── 干员编队（1-12人）                          │
│  ├── 单人/多人组队（/team + UI 邀请）             │
│  └── 开始作战 → CombatSessionManager            │
└─────────────────────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│  关卡维度（代码注册超平坦）                       │
│                                                 │
│  CombatSessionManager (全局单例)                │
│  ├── 固定槽位池 (BitSet 占用标记)                │
│  ├── 512² 分配区域 / 256³ 关卡区域               │
│  └── CombatSession (作战会话)                   │
│       ├── 实例级门映射 (Map<Integer, BlockPos>)  │
│       ├── 控制器 BE 引用                         │
│       ├── 区域 AABB + 玩家列表                   │
│       ├── 词条上下文 → 系统层 + 节点图注入        │
│       └── IDLE → RUNNING → VICTORY/DEFEAT      │
└─────────────────────────────────────────────────┘
```

### 1.2 设计原则

- **编辑器 = 设计工具，控制台 = 游戏入口**。关卡在编辑器中搭好，在控制台中被选中并运行。
- **一个关卡 = 一个控制器 = 一个节点图**。
- **关卡文件 = StructureTemplate NBT + 自定义元数据扩展根节点**。创作者可分发 `.nbt` 文件。
- **共享维度 + 固定槽位偏移**实现多人同时跑不同关卡。
- **关卡结束后批量清理**方块和实体，释放槽位。

## 二、关卡文件格式

### 2.1 文件结构

基于 Minecraft 原版 `StructureTemplate` NBT 格式，扩展自定义根节点：

```
关卡文件 (.nbt)
├── palette  (ListTag)      ← 原版方块调色板
├── blocks   (ListTag)      ← 原版方块位置索引列表
├── size     (ListTag)      ← [x, y, z] 尺寸
├── entities (ListTag)      ← 原版实体
├── ModularDefendMeta       ← 自定义：关卡元数据
│   ├── name        (String)   关卡名
│   ├── author      (String)   作者
│   └── doors       (Compound) 门 ID 映射
│       ├── redDoors  { "1": [x,y,z], "2": [x,y,z], ... }
│       └── blueDoors { "1": [x,y,z], "2": [x,y,z], ... }
└── NodeGraphData           ← 自定义：控制器节点图 CompoundTag
```

### 2.2 文件管理

- **存储位置**: `config/modularization_defend/levels/`
- **文件后缀**: `.nbt`
- **文件命名**: 默认用关卡名，非法字符（特殊符号）替换为 `"new"`
- **发现方式**: 服务端启动/重载时扫描目录，构建缓存索引
- **同名覆盖**: 保存前弹出确认提示（复用 LDLib2 已有实现）

### 2.3 元数据来源

- **关卡名/作者**: 由设计者在编辑器 UI 中输入，存储在 `ModularDefendMeta`
- **门映射**: 保存时自动扫描区域内所有 `RedDoorBlockEntity` / `BlueDoorBlockEntity`

## 三、编辑器验证规则

### 3.1 保存时验证（两层：编辑器 + 控制台）

| 检查项 | 编辑器行为 | 控制台行为 |
|---|---|---|
| 没有控制器 | **拒绝保存**，提示放置控制器 | **拒绝启动** |
| 门未设置 ID | 警告，确认后**自动分配递增 ID** | **拒绝启动** |
| 节点图引用了不存在的门 ID | **拒绝保存** | — |
| 节点图为空 | 警告，确认后允许保存 | **拒绝启动** |
| 没有红门或没有蓝门 | 警告，确认后允许保存 | **拒绝启动** |
| 文件损坏/格式错误 | — | **拒绝启动**，提示创作者 |

## 四、关卡运行维度

### 4.1 维度定义

- **类型**: 代码注册的超平坦维度（`FlatLevelSource`）
- **隔离方式**: 固定槽位池，每个槽位分配坐标偏移
- **移动限制**: 512x512（每槽位分配范围），**不限高**
- **建造区域**: 256x256x256（实际关卡结构放置范围）

### 4.2 槽位管理

```
槽位 0:  偏移 (0, 0)      ← BitSet bit 0
槽位 1:  偏移 (512, 0)    ← BitSet bit 1
槽位 2:  偏移 (1024, 0)   ← BitSet bit 2
...
```

- 使用 `BitSet` 标记占用/空闲
- 关卡未完全清理前该区域标记为**占用**
- 放置时如果遇到残留方块，**直接覆盖**（残留不应发生，覆盖作为兜底）

### 4.3 坐标变换

- **编辑器原点**: 建造区域的**最小角**（而非编辑器方块本身）
- **放置坐标**: `editorRelativePos + slotOffset`
- 结构中的方块实体 `BlockPos` 通过 `StructureTemplate.placeInWorld()` 自动偏移

## 五、CombatSession 生命周期

### 5.1 会话创建流程

```
1. CombatSessionManager.assignSlot() → 找到空闲槽位
2. 读取关卡 .nbt 文件
3. StructureTemplate.load() + placeInWorld()
4. 扫描粘贴后的门 BE → 构建实例级 doorMap
5. 找到唯一 LevelControllerBE → 获取引用
6. CombatSessionManager.createSession(slot, ...)
7. LevelControllerBE.startLevel()
8. 传送所有参与玩家到维度中
```

### 5.2 会话状态机

```
IDLE → RUNNING → VICTORY (setLevelResult(true))
               → DEFEAT  (setLevelResult(false))
               → SUSPENDED (所有玩家离线/主动暂停)
```

### 5.3 暂停与恢复

- **触发条件**: 所有玩家离线 / 玩家主动暂停（通过干员部署 UI）
- **行为**: 冻结区域内所有实体 AI，保留所有运行时状态
- **恢复**: 玩家上线确认后解冻

### 5.4 退出与投票

- **单人退出**: 干员部署 UI 中的退出按钮，直接结束
- **多人退出**: 发起投票，通过后退出一人，其他玩家继续
- **全员退出**: 会话标记结束

### 5.5 掉线处理

- 玩家掉线 → 标记为掉线状态
- 恢复上线 → 如果在活跃会话中则恢复，如果关卡已结束则**传送回重生点**

### 5.6 服务器关停

- `ServerStoppingEvent` → 所有活跃会话**冻结状态 → 保存到磁盘**
- 服务器重启 → 检测未恢复会话，走 5.3 的恢复流程

### 5.7 清理

- 分批清理（`Level.scheduleTick` 分帧）：
  - 每 tick 清理 N 个方块位置（设为 `Blocks.AIR`）
  - `entity.discard()` 所有追踪实体
- 全部清完后标记槽位为空闲

## 六、门系统

### 6.1 门类型

| 类型 | 方块 | 功能 |
|---|---|---|
| 红门 (RedDoor) | 3x3x3 多方块 | 出怪口，敌人生成点 |
| 蓝门 (BlueDoor) | 3x3x3 多方块 | 防守点，漏怪判定区 |

### 6.2 门 ID

- **ID 类型**: 数字（1~N）
- **分配方式**: 设计者在编辑器中使用**手持工具**右键门主方块设置
- **红门和蓝门的 ID 可以相同**（类型不同，不会冲突）

### 6.3 门注册

- **当前**: `DoorRegistry` 静态工具类（`findDoor` 扫描世界中所有门 BE）
- **计划**: 迁移为 `CombatSession` 实例级 `Map<Integer, BlockPos>`
- **原因**: 共享维度 + 多关卡同时运行时，静态注册表会导致不同关卡的门 ID 冲突

## 七、词条系统

### 7.1 词条池

- **全局词条池**: 所有关卡通用的词条（如"敌方攻击力+20%"）
- **关卡覆盖**: 关卡可追加/替换部分词条，创建自定义组合

### 7.2 生效方式（混合模式）

```
词条"敌方攻击力+20%"
├── 系统层: 给实体加 20 级攻击力加成词条（直接修改属性）
└── 节点图层: 注入 trait:enemyBuff=0.2 到外部上下文（图内条件节点可响应）
```

### 7.3 玩家视图

- 玩家只看到效果描述（如"敌方攻击力+20%"），不关心后端实现
- 词条前端展示通过控制台 UI XML 管理

## 八、关卡编辑器

### 8.1 尺寸

- **可建造区域**: 256x256x256
- **坐标原���**: 区域最小角

### 8.2 保存流程

```
saveZone():
  1. 遍历区域 → StructureTemplate.fillFromWorld()
  2. 扫描门 BE → 收集 {id → (x,y,z)} 映射
  3. 找到控制器 BE → 提取 graphTag
  4. 验证（见第三节）
  5. template.save() + 附加 ModularDefendMeta + NodeGraphData
  6. 写入 config/modularization_defend/levels/<name>.nbt
```

### 8.3 加载流程

```
loadZone():
  1. 读取关卡文件 → StructureTemplate.load()
  2. 清空当前区域
  3. template.placeInWorld()
  4. 门 BE 自动创建（含 doorId），控制器 BE 自动创建（含 graphTag）
```

### 8.4 清空

```
clearZone():
  1. 遍历区域 → 所有非空气方块设为 AIR
  2. 移除所有实体
  3. 提示完成
```

## 九、作战控制台

### 9.1 模式选择

| 模式 | 说明 |
|---|---|
| **危机合约** | 自选词条难度，挑战高分 |

- 在 UI 中以列表展示

## 十、玩家与干员

### 10.1 干员

- **类型**: 自定义 `LivingEntity`，GeckoLib 动画
- **部署**: 选中可部署区 → 放置干员 → 扣除部署费用
- **行为**: 自动攻击范围内敌人（或治疗友方）
- **技能**: 自动充能充能，充能完成后自动触发；部分技能需手动触发
- **数据存储**: 玩家持久化数据（`PlayerData`），关卡作者可选覆盖
- **被击败**: 进入再部署冷却（非永久阵亡）
- **部署限制**: 每格可部署区限 1 个干员
- **移动**: 干员部署后不移动

### 10.2 玩家状态

- 部署干员后切换为**旁观模式**
- 旁观模式下大部分伤害免疫（意外死亡则走原版重生机制，通过控制台重新加入）
- **常驻 HUD** 显示干员栏 + 费用条 + 技能按钮
- 按键**切换释放光标**（切换 HUD 交互 / 世界交互）

### 10.3 干员职业（参考）

| 职业 | 部署位 | 特点 |
|---|---|---|
| 先锋 | 地面 | 低费、回费 |
| 近卫 | 地面 | 近战输出 |
| 重装 | 地面 | 高防、阻挡 |
| 狙击 | 高台 | 远程物理输出 |
| 术师 | 高台 | 远程法术输出 |
| 医疗 | 高台 | 治疗友方 |
| 辅助 | 高台 | 减益/控制 |
| 特种 | 地面/高台 | 特殊机制 |

> 干员数据模型由其他成员负责，不在此文档范围内。

## 十一、节点图系统

### 11.1 双引擎架构

```
┌──────────────────────────────────┐
│  LevelGraphRuntime (tick 驱动)    │
│  ┌────────────┐  ┌────────────┐ │
│  │ 数据流引擎  │  │ 执行流引擎  │ │
│  │ Evaluator  │  │ ExecEngine │ │
│  │ (拉取式)    │  │ (推送式)    │ │
│  └─────┬──────┘  └─────┬──────┘ │
│        │ 注入外部上下文   │ 边沿检测 │
│        ▼                ▼        │
│  数据流节点求值    事件条件检查    │
│  (递归追溯连线)    (上升沿触发)    │
│                      │           │
│                      ▼           │
│                动作链遍历执行      │
│                (深度上限 64)      │
└──────────────────────────────────┘
```

### 11.2 图编辑器

- `/nodegraph empty` — 打开空白图编辑器
- `/nodegraph` — 打开含演示节点的编辑器
- `/nodegraph editor` — 打开带资源面板的资源编辑器（调试用）
- 潜行右键控制器 — 打开嵌入式编辑器（数据自动存入控制器 BE）

## 十三、已知技术债务

- LDLib2 `NodeOption.getDataType()` 恒返回 `null`，`NodeGraphEvaluator.getOptionValue()` 需从底层 `PortModel` 的 `TypeHandle` 解析类型（已修复，勿回退）
- 求值器/执行器使用中文 `@NodeAttribute.name()` 字符串注册，注册表 key 与节点类名不一致，易遗漏同步
- 新增关卡节点需三处同步：节点类 + `LevelNodeExecutors`/`NodeEvaluators` 注册 + zh_cn/en_us 语言文件
